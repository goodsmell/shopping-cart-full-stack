# 장바구니 풀스택 미션 — 3단계 요구사항

---

## 1. 상태 설계

### 핵심 원칙

- 상품 정보, 수량, 선택 여부, 쿠폰, 배송 옵션은 하나의 원천 상태를 기준으로 관리한다.
- 이번 설계에서는 핵심 상태의 원천을 **서버 DB**로 둔다.
- 프론트엔드는 서버 상태를 조회해 화면에 렌더링한다.
- 프론트엔드는 결제 금액, 할인 금액, 배송비, 최종 결제 금액을 **직접 계산하지 않는다.**
- 금액 계산은 **항상 서버가 수행**한다.

---

### 상태별 책임

| 상태 | 원천 | 설명 |
|------|------|------|
| 상품 정보 | 서버 DB | 상품명, 가격, 이미지, 재고 |
| 장바구니 수량 | 서버 DB | 사용자가 장바구니에 담은 상품 수량 |
| 상품 선택 여부 | 서버 DB | 주문 대상으로 선택한 장바구니 항목 |
| 쿠폰 정보 | 서버 DB | 쿠폰 정책과 만료일 |
| 선택한 쿠폰 | 서버 DB | 주문 확인 상태에 적용된 쿠폰 |
| 도서산간 선택 여부 | 서버 DB | 주문 확인 상태의 배송 옵션 |
| 주문 금액 | 서버 계산값 | 선택된 상품 가격과 수량 기준 |
| 할인 금액 | 서버 계산값 | 쿠폰 정책 기준 |
| 배송비 | 서버 계산값 | 주문 금액, 도서산간 여부, 무료배송 쿠폰 기준 |
| 최종 결제 금액 | 서버 계산값 | 주문 금액 - 할인 금액 + 배송비 |

---

### 장바구니 상태 관리

- 장바구니 페이지 진입 시 서버에서 장바구니 상품 목록을 조회한다.
- 사용자가 상품을 선택하거나 선택 해제하면 서버에 선택 상태 변경 요청을 보낸다.
- 사용자가 수량을 변경하면 서버에 수량 변경 요청을 보낸다.
- 사용자가 상품을 삭제하면 서버에 삭제 요청을 보낸다.
- 장바구니 변경 후 결제 정보는 서버가 다시 계산한다.

---

### 주문 확인 상태 관리

- 주문 확인은 **최종 주문이 아니다.**
- 주문 확인은 사용자가 최종 주문 전 상품, 쿠폰, 배송비, 결제 금액을 확인하는 임시 상태이다.
- 사용자가 장바구니에서 "주문 확인" 버튼을 클릭하면 서버는 `order-check`를 생성한다.
- `order-check` 생성 시 클라이언트는 **선택된 상품 목록을 body로 전달하지 않는다.**
- 서버는 현재 사용자의 장바구니에서 `selected = true`인 항목을 조회하여 주문 확인 정보를 생성한다.
- 생성된 주문 확인 상태는 `orderCheckId`로 식별한다.

## 2. 쿠폰

### 2.1 공통 조건

- 쿠폰 데이터는 서버 DB에 미리 저장되어 있다.
- 쿠폰 등록, 수정, 삭제 API는 구현하지 않는다.
- 사용자는 주문 확인 페이지에서 쿠폰을 조회하고 선택할 수 있다.
- 쿠폰 적용 가능 여부와 할인 금액은 서버가 계산한다.
- 프론트엔드는 서버가 계산한 쿠폰 정보를 화면에 표시한다.

---

### 2.2 쿠폰 목록

| 쿠폰 ID | 쿠폰명 | 조건 | 혜택 | 만료일 |
|---------|--------|------|------|--------|
| `FIXED5000` | 5,000원 할인 쿠폰 | 최소 주문 금액 100,000원 | 5,000원 할인 | 2026-11-30 |
| `BOGO` | 2+1 쿠폰 | 동일 상품 2개 이상 구매 | 단가가 가장 높은 상품 1개 무료 | 2026-06-30 |
| `FREESHIPPING` | 무료 배송 쿠폰 | 최소 주문 금액 50,000원 | 기본 배송비와 도서산간 추가 배송비 무료 | 2026-08-31 |
| `MIRACLESALE` | 30% 시간제 할인 쿠폰 | 오전 4시 ~ 7시 | 30% 할인 | 2026-07-31 |

---

### 2.3 쿠폰별 상세 정책

**FIXED5000**
- 최소 주문 금액이 100,000원 이상이어야 한다.
- 조건을 만족하면 주문 금액에서 5,000원을 할인한다.
- 만료일 이후에는 사용할 수 없다.

**BOGO**
- 동일 상품을 2개 이상 구매한 경우 사용할 수 있다.
- 조건을 만족하는 상품 중 단가가 가장 높은 상품 1개 가격만큼 할인한다.
- 여러 상품이 조건을 만족하면 서버가 가장 할인 효과가 큰 상품을 선택한다.
- 만료일 이후에는 사용할 수 없다.

**FREESHIPPING**
- 최소 주문 금액이 50,000원 이상이어야 한다.
- 기본 배송비를 무료 처리한다.
- 도서산간 추가 배송비도 무료 처리한다.
- 만료일 이후에는 사용할 수 없다.

**MIRACLESALE**
- 오전 4시부터 오전 7시 사이에만 사용할 수 있다.
- 할인율은 30%이다.
- 정액 쿠폰과 함께 적용되는 경우, 정액 할인을 먼저 적용한 뒤 남은 금액에 30%를 적용한다.
- 만료일 이후에는 사용할 수 없다.

## 3. 쿠폰 복합 적용

### 3.1 최대 선택 개수

- 사용자는 최대 **2개**의 쿠폰을 동시에 사용할 수 있다.
- 2개를 초과해 쿠폰 적용 요청을 보내면 서버는 에러를 반환한다.
- 서버에서 에러를 반환하므로, 클라이언트에서 클릭을 막는 등의 별도 처리가 필요하다.

### 3.2 할인 계산 순서

1. 정액 할인 쿠폰을 먼저 적용한다.
2. 정액 할인이 적용된 금액을 기준으로 정율 할인 쿠폰을 적용한다.
3. 배송비 쿠폰은 배송비 계산 단계에서 적용한다.
4. 최종 결제 금액을 계산한다.

### 3.3 가장 할인 효과가 큰 조합

- 서버는 현재 주문 확인 상태에서 가장 할인 효과가 큰 쿠폰 조합을 계산할 수 있어야 한다.
- 쿠폰 모달에서는 쿠폰별 예상 할인 금액을 보여줄 수 있다.
- 프론트엔드는 서버가 내려준 예상 할인 금액을 화면에 표시할 수 있다.
- 실제 최종 할인 금액은 서버가 계산한다.

## 4. 쿠폰 기능

### 4.1 사용 가능한 쿠폰 목록 조회

사용자는 주문 확인 페이지에서 사용 가능한 쿠폰 목록을 확인할 수 있어야 한다.
쿠폰 목록 조회 시 서버는 현재 주문 확인 상태를 기준으로 쿠폰 정보를 계산한다.

서버가 판단하는 기준:
- 현재 주문 금액
- 주문 대상 상품 및 상품별 수량
- 쿠폰 만료일 / 쿠폰 최소 주문 금액
- 현재 시각
- 도서산간 선택 여부 / 배송비 정책
- 쿠폰 조합 가능 여부

### 4.2 쿠폰 목록 응답 정보

쿠폰 목록에는 다음 정보가 포함될 수 있다.

- 쿠폰 ID / 쿠폰명 / 쿠폰 설명
- 사용 가능 여부 / 사용 불가능 사유
- 예상 할인 금액
- 만료일 / 최소 주문 금액

### 4.3 쿠폰 선택 UI

- 사용자는 쿠폰 모달에서 쿠폰을 선택할 수 있다.
- 최대 2개까지 선택할 수 있다.
- 사용 불가능한 쿠폰은 비활성화 상태로 표시한다.
- 쿠폰별 예상 할인 금액을 표시한다.
- 가장 할인 효과가 큰 쿠폰 조합을 기본 선택 상태로 보여줄 수 있다.

### 4.4 쿠폰 적용

1. 사용자가 "쿠폰 사용" 버튼을 누르면 선택한 쿠폰 목록을 서버에 전달한다.
2. 서버는 선택된 쿠폰의 유효성을 검증한다.
3. 서버는 할인 금액, 배송비, 최종 결제 금액을 다시 계산한다.
4. 서버는 계산된 결과를 주문 확인 상태에 반영한다.
5. 프론트엔드는 서버 응답값으로 주문 확인 화면을 갱신한다.

### 4.5 쿠폰 적용 실패 조건

쿠폰 적용 요청은 다음 경우 실패할 수 있다.

- 선택한 쿠폰이 2개를 초과한 경우
- 존재하지 않는 쿠폰인 경우
- 만료된 쿠폰인 경우
- 최소 주문 금액을 만족하지 않는 경우
- 시간제 쿠폰의 사용 가능 시간이 아닌 경우
- 현재 주문 상품에 적용할 수 없는 쿠폰인 경우

## 5. 배송비 정책

### 5.1 기본 배송비

- 기본 배송비는 **3,000원**이다.
- 배송비는 서버에서 계산한다.
- 무료 배송 기준은 **쿠폰 적용 전 주문 금액**이다.
- 주문 금액이 100,000원 이상이면 기본 배송비는 무료이다.
- 주문 금액이 100,000원 미만이면 기본 배송비를 부과한다.

### 5.2 도서산간 추가 배송비

- 제주도 및 도서산간 지역이 선택되면 배송비에 **3,000원**을 추가한다.
- 도서산간 추가 배송비는 주문 확인 페이지의 체크박스 상태에 따라 결정된다.
- 체크 상태는 서버의 주문 확인 상태에 저장한다.

### 5.3 무료 배송 쿠폰 적용

- `FREESHIPPING` 쿠폰이 적용되면 기본 배송비를 무료 처리한다.
- `FREESHIPPING` 쿠폰은 도서산간 추가 배송비도 무료 처리한다.
- 따라서 도서산간 지역이 선택되어 있어도 `FREESHIPPING` 쿠폰이 적용되면 배송비는 0원이 될 수 있다.

### 5.4 배송비 계산 순서

1. 쿠폰 적용 전 주문 금액을 계산한다.
2. 주문 금액이 100,000원 이상인지 확인한다.
3. 기본 배송비를 계산한다.
4. 도서산간 선택 여부를 확인한다.
5. 도서산간 추가 배송비를 계산한다.
6. `FREESHIPPING` 쿠폰 적용 여부를 확인한다.
7. 최종 배송비를 계산한다.

## 6. 배송비 정책 UI

### 6.1 도서산간 체크박스

주문 확인 페이지에는 다음 체크박스를 제공한다.

> ☐ 제주도 및 도서산간 지역

### 6.2 체크 시 동작

1. 프론트엔드는 서버에 도서산간 선택 상태 변경 요청을 보낸다.
2. 서버는 주문 확인 상태의 도서산간 여부를 갱신한다.
3. 서버는 배송비와 최종 결제 금액을 다시 계산한다.
4. 서버는 갱신된 금액 정보를 응답한다.
5. 프론트엔드는 응답값으로 화면을 갱신한다.

### 6.3 체크 해제 시 동작

1. 프론트엔드는 서버에 도서산간 선택 해제 요청을 보낸다.
2. 서버는 도서산간 추가 배송비를 제거한다.
3. 서버는 배송비와 최종 결제 금액을 다시 계산한다.
4. 프론트엔드는 서버 응답값으로 화면을 갱신한다.

### 6.4 요청 방식

도서산간 선택은 **토글 방식으로 처리하지 않는다.**
프론트엔드는 변경하려는 값을 명확히 전달한다.

```json
{ "isRemoteArea": true }
// 또는
{ "isRemoteArea": false }
```

> 같은 요청을 여러 번 보내도 같은 상태가 유지된다. (멱등성 보장)

## 7. FE / BE 책임 정리

### 7.1 Frontend 책임

**담당하는 것**
- 장바구니 상품 목록 렌더링
- 주문 확인 페이지 렌더링
- 쿠폰 모달 렌더링
- 사용자 입력 감지 (체크박스 선택, 쿠폰 선택)
- 서버 요청 전송
- 서버 응답값 화면 반영

**담당하지 않는 것**
- 할인 금액 직접 계산
- 배송비 직접 계산
- 최종 결제 금액 직접 계산
- 쿠폰 유효성 최종 판단

### 7.2 Backend 책임

**담당하는 것**
- 장바구니 상태 관리
- 상품 선택 상태 저장
- 주문 확인 상태 생성
- 쿠폰 사용 가능 여부 판단
- 쿠폰 할인 금액 계산
- 배송비 계산
- 최종 결제 금액 계산
- 주문 생성 전 최종 검증
